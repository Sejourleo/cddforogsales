# ToB 客户拜访 AI 背调

## Overview

这个 skill 用来在拜访大客户、KA 客户、渠道客户或潜在客户前，快速把公开信息转化成**可验证的销售判断**。

核心原则：背调不是收集信息，也不是尽调；背调的目标是形成一组假设，让销售在第一次见面时能说对第一句话，并设计出高质量验证问题。

默认产出不是“百度百科式公司介绍”，而是一页可用于拜访前准备的报告：

- 这家公司大概处在什么阶段？
- 所在行业最近发生了什么变化？
- 如果采购我们的产品，决策链可能怎么走？
- 对接人可能关心什么 KPI？
- 我们应该从哪个角度切入？
- 哪些信息是事实，哪些只是推测，见面时该怎么验证？

## When to Use

使用场景：

- 用户说“帮我做某客户背调 / 客户拜访准备 / 大客户拜访准备 / 销售拜访准备”。
- 用户给出公司名、联系人、职位、行业、会议目标，想要一份拜访前报告。
- 用户想为首次见面、售前 discovery call、渠道拜访、BD 约谈准备破冰话术。
- 用户想把公司公开资料转成“痛点假设 + 第一句话 + 验证问题”。

不要用于：

- 法务、投融资、并购等正式尽调。
- 非公开个人隐私调查。
- 把未经验证的 AI 推测包装成事实。

## Minimum Inputs

尽量从用户提供的信息直接开始；不要为了完美信息反复追问。

最低输入：

- 客户公司名。
- 用户要销售/推荐的产品或方案。

有则更好：

- 行业或业务线。
- 对接人姓名、职位、部门。
- 会议目标：初次拜访、复盘、推进预算、技术交流、方案汇报等。
- 已知线索：中间人评价、客户最近动作、官网/新闻链接、用户自己的判断。

如果用户没有说明自己的产品/方案，且上下文没有其它说明，默认只围绕 OGCLOUD 产品范围做销售拜访准备：SD-WAN 加速、SD-WAN 组网、IDC、大带宽+DDoS、MSP、网络安全、SASE 零信任解决方案。

**OGCLOUD 业务范围规则：**必须把客户背调、痛点假设、决策链猜测、开场话术和验证问题都收敛到 OGCLOUD 相关产品。不要引入非 OGCLOUD 业务；除非客户公开信息强烈显示网络、安全或基础设施需求，也只作为“待验证问题”，不要包装成既定事实。

## Research Rules

For Chinese private-company/prospect lookup, use the tactical checklist in `references/chinese-company-lookup-tactics.md` when registry sites are sparse, anti-bot protected, the company appears newly established, or the official site is a modern Nuxt/Vue app whose visible HTML is thin (the reference includes a JSONP/API-route extraction pattern for such sites).

1. **只使用公开信息**：官网、新闻稿、招聘、公开演讲、公众号、招股书、年报、工商摘要、行业报告等。
2. **区分三类内容**：
   - 已验证事实：有来源或用户提供。
   - 合理推测：基于行业/角色/公司阶段推断。
   - 待验证问题：见面时要问客户确认。
3. **人物信息高风险**：联系人履历、观点、前任公司、公开发言必须标注来源；查不到就写“公开信息不足”，不要补全。
4. **时效性强的信息要复核**：融资、并购、高管变动、业务调整、政策变化、重大客户签约等，必须看时间。
5. **不要断言客户内部情况**：痛点、预算、决策链都用“我猜 / 可能 / 值得验证”的口吻。
6. **目标是生成销售动作**：每个判断都要能落到一句开场话术、一个验证问题、或一个下一步动作。

## 10 分钟背调 SOP

### Step 1：公司基本面 → 建立整体画像

目标：快速建立对客户的“体感”，不是罗列工商信息。

关注：

- 主营业务和商业模式。
- 客户类型、收入来源、核心产品/服务。
- 当前规模量级：营收、人数、融资阶段、区域布局、客户群。
- 最近 1-2 年重大变化：融资、并购、业务调整、高管变动、出海、组织扩张/收缩。

判断句模板：

> 这家公司不是单纯的 [表面分类]，更像是一家处在 [阶段] 的 [业务本质] 公司；最近的关键变化是 [变化]，这可能意味着 [销售相关含义]。

### Step 2：行业痛点和趋势 → 准备“抛判断”的弹药

目标：让客户感觉你不是只查了他家公司，而是懂他的行业语境。

关注：

- 最近一年政策/监管变化。
- 技术趋势和替代路径。
- 竞争格局变化。
- 成本、效率、增长、合规、出海、数据、安全等业务压力。
- 这些变化对业务负责人意味着什么。

输出要从业务决策者视角写，不要陷入技术名词堆砌。

### Step 3：决策链猜测 → 知道该问什么

目标：预测采购相关角色和预算路径，帮助销售设计探询问题。

关注：

- 使用部门、需求发起人、技术评估人、采购/法务/财务、最终拍板人。
- 预算来自哪里：业务预算、IT 预算、市场预算、专项项目预算、总部预算。
- 常见阻力：ROI 不清、集成成本、合规安全、内部资源不足、老板优先级不够。

注意：这是行业通用推测，不是该公司的内部事实。

### Step 4：对接人画像 → 判断对方可能关心什么

目标：理解对接人的角色目标和 KPI，而不是八卦个人隐私。

关注：

- 公开可验证的职业背景。
- 公开发言、文章、采访、社交资料。
- 当前职位对应的 KPI：增长、降本、效率、合规、安全、交付、团队管理等。
- 这个人可能喜欢什么沟通方式：战略视角、数据证据、落地案例、风险控制、快速试点。

如果公开信息不足，直接写：

> 公开信息不足，不能可靠判断其个人履历；以下只基于其职位角色推测可能关注点。

### Step 5：切入点预判 → 设计第一句话

目标：形成一个能被客户纠正的具体判断。

好的开场不是“我看到贵司成立于 2018 年”，而是：

> 我注意到你们最近在 [业务变化/行业趋势]，我猜现在比较棘手的可能不是 [表层问题]，而是 [更深层问题]。这个判断不一定准，我想先听听你们实际情况。

## Prompt Pack

以下是文章中沉淀出的 5 个核心 prompt，可单独使用，也可以作为报告生成链条。

### 1. 公司基本面

```text
请帮我分析[公司名]这家公司。我需要了解：
1）他们的主营业务和商业模式是什么；
2）目前的规模大概在什么量级（营收/人数/融资阶段）；
3）最近 1-2 年有没有什么重大变化（融资、并购、业务调整、高管变动）。
请用简洁的语言概括，不要罗列工商信息。
```

### 2. 行业痛点和趋势

```text
[公司名]所在的[行业]，最近一年有哪些重要的政策变化、技术趋势或竞争格局变化？
这些变化可能给这类企业带来什么挑战或机会？
请从业务决策者的视角来分析，不要太技术化。
```

### 3. 决策链猜测

```text
像[公司名]这种规模的[行业]企业，如果要采购[你的产品类型]，通常的决策链条是什么样的？
可能涉及哪些角色（IT 负责人/业务负责人/财务/CEO）？
最终拍板的通常是谁？预算审批一般走什么流程？
```

### 4. 对接人画像

```text
请帮我了解[人名]，他目前是[公司名]的[职位]。
我想知道：
1）他的职业背景和履历；
2）他公开发表过什么观点或文章；
3）基于他的角色，他最可能关心什么 KPI 或业务指标？
请基于公开信息分析，如果信息有限请明确告诉我。
```

### 5. 切入点预判

```text
基于以上分析，我是一家[你的公司简介]的销售。
我们的产品主要解决[核心价值]。
请帮我判断：
1）这家客户最可能的痛点是什么；
2）我的产品能帮他解决什么具体问题；
3）见面时我应该从什么角度切入话题，才能让对方觉得我懂他。
```

## One-Shot Master Prompt

当用户想复制到其它 AI 工具里，可以给这个整合版：

```text
你是一名资深 ToB 大客户销售顾问。请基于公开信息和合理行业判断，帮我为一次客户拜访做会前背调。

已知信息：
- 客户公司：[公司名]
- 所属行业：[行业]
- 对接人：[姓名/职位，如未知写未知]
- 我方公司/方案：[你的公司简介]
- 我方核心价值：[核心价值]
- 拜访目标：[初次拜访/需求挖掘/推进预算/方案汇报等]

请输出：
1. 公司基本面：主营业务、商业模式、规模阶段、最近 1-2 年重大变化。不要罗列工商信息。
2. 行业痛点和趋势：最近一年政策、技术、竞争格局变化，以及对业务决策者的影响。
3. 决策链猜测：如果采购我方方案，通常涉及哪些角色、谁可能拍板、预算审批可能怎么走。
4. 对接人画像：只基于公开信息分析其履历、观点、可能 KPI；信息不足请明确说明，不要编造。
5. 切入点预判：客户最可能的痛点、我方能解决的具体问题、见面时建议从哪个角度切入。
6. 开场第一句话：用“我注意到...我猜...这个判断不一定准，想听听你们实际情况”的形式写 2-3 个版本。
7. 5 个验证问题：用于会中验证你的判断。
8. 信息边界：把内容分为“已验证事实 / 合理推测 / 待验证问题”。

要求：
- 不要写百科式介绍。
- 不要把推测说成事实。
- 对人物信息尤其谨慎；没有可靠公开来源就写信息不足。
- 输出要简洁、可用于销售拜访。
```

## Opening-Line Patterns

### 趋势切入

> 我注意到最近 [行业变化] 对 [客户类型] 影响挺大，我猜你们现在可能也在看 [具体问题]。这个判断不一定准，想听听你们实际情况。

### 组织变化切入

> 我看到你们最近在 [融资/出海/扩张/平台化/组织调整]，我猜现在最难的可能不是 [表层问题]，而是 [底层问题]。

### KPI 切入

> 按您这个角色，我猜今年更核心的指标可能是 [KPI]，所以我今天不想先讲产品功能，想先确认一个问题：[验证问题]。

### 风险切入

> 很多 [行业] 企业做到这个阶段后，都会遇到 [风险/瓶颈]。我不确定你们是不是也类似，所以想先验证一下。

## Common Pitfalls

1. **把 AI 输出当事实。** 正确做法是把输出当假设，用于会中验证。
2. **人物信息幻觉。** AI 很容易编造联系人履历、前公司、观点。没有来源就不要在会中提。
3. **信息过期。** 融资、高管、业务调整、政策变化要看时间。
4. **开场太装。** 不要说“我知道你们现在最大问题是...”，要说“我猜...不一定准”。
5. **堆行业术语。** 客户买的是业务结果，不是术语密度。
6. **只查客户，不映射我方价值。** 每个痛点假设都必须连接到我方能帮什么，以及下一步怎么验证。
7. **过度私人化。** 对接人画像只用于理解角色目标，不做隐私窥探。

## Verification Checklist

- [ ] 公司名、行业、拜访目标明确。
- [ ] 我方产品/方案和核心价值明确；未提供时已使用默认 Yves 出海 AI/IT 方案。
- [ ] 报告里区分了“事实 / 推测 / 待验证”。
- [ ] 人物信息没有无来源断言。
- [ ] 近期变化标注了时间或来源。
- [ ] 至少有 1 条具体开场话术。
- [ ] 至少有 5 个会中验证问题。
- [ ] 结论能支持下一步销售动作，而不是停留在信息介绍。
